# Pipeline Audit Governance Design

Date: 2026-05-25

## Context

The 2026-05-25 daily report exposed a governance failure, not just a data gap.
Stage3 was eventually run with a debug-only combination:

- `--allow-estimated`
- `--allow-fallback`
- `--skip-fund-flow-check`
- `--skip-gap-check`

That produced `pring_result.json` with `fallback_used=true`. Stage4 was then
generated by calling `generate_report()` directly, bypassing the formal
Stage4 checks for unified quality, gap monitor state, and Pring date
consistency.

The immediate trigger was `fund_flow.etf` remaining unresolved and BDI being
blocked by a calendar-day freshness rule on a Monday morning. The deeper
problem is that quality checks are spread across several modules and do not
all expose the same effective gate semantics:

- `build_pipeline_quality_state()` is the intended shared quality-state core.
- Stage3 has `--skip-fund-flow-check`.
- Stage4 does not yet have equivalent fund-flow skip semantics.
- `simple_report` still computes a report quality section with its own logic.
- Stage2.5 has legacy `_enforce_quality_blockers()` tests that can drift from
  the shared quality state.
- Manual evidence validation checks that a URL exists, but not whether the URL
  actually supports the value or matches the claimed source.

## Goals

1. Make Stage2.5, Stage3, Stage4, and the report quality section consume one
   coherent quality language.
2. Keep production report generation inside `scripts/stage4_report_generator.py`.
3. Allow a production-grade fund-flow-only downgrade when ETF window data is
   unavailable, without allowing broader gate bypass.
4. Prevent fallback Stage3 results from becoming formal reports by default.
5. Replace the BDI Monday failure mode with explicit business-day freshness
   semantics.
6. Add audit tooling that detects rule drift across stages before a report is
   generated.
7. Add manual evidence checks for source URL provenance without redefining the
   existing estimated, official-source, and fund-flow window rules.
8. Update `AGENTS.md` and `CLAUDE.md` so the operational runbook matches the
   implementation.
9. Design the implementation so multiple agents can work in parallel with
   clear ownership boundaries.

## Non-Goals

1. Do not relax non-fund-flow Stage3 or Stage4 blockers.
2. Do not let `--allow-estimated` bypass compare gaps, stale critical fields,
   policy blockers, or non-allowlisted estimates.
3. Do not make `--skip-gap-check` a production option.
4. Do not make `--allow-fallback` or fallback reports production defaults.
5. Do not use generated `reports/*.md` as data sources.
6. Do not promote ETF news, quarterly summaries, year-to-date summaries,
   single-day extrapolations, or `estimated_net_flow` into trusted fund-flow
   windows.
7. Do not duplicate Stage2.5 official override or fund-flow tier/window rules
   inside the audit layer.

## Chosen Approach

Use a shared quality-state model plus thin stage-specific adapters:

```text
market_data_complete.json
  -> build_pipeline_quality_state()
  -> stage effective blocker filters
      -> Stage2.5 artifacts
      -> Stage3 gate
      -> Stage4 gate
      -> report quality section
  -> audit consistency checks
  -> manual evidence audit
```

The audit tools explain and compare existing rules. They do not become a new
independent gate vocabulary.

## Production Gate Semantics

### Stage3

The production command may use:

```bash
bash run_clean.sh python scripts/stage3_pring_analyzer.py \
  --market-data "data/runs/${DATE_NH}/market_data_complete.json" \
  --output "data/runs/${DATE_NH}/pring_result.json" \
  --allow-estimated \
  --skip-fund-flow-check
```

Allowed meaning:

- `--allow-estimated` lets allowlisted estimated values participate in scoring.
- `--skip-fund-flow-check` filters only fund-flow quality blockers whose
  reasons are explicitly fund-flow window or fund-flow estimate reasons.

Still blocked:

- non-fund-flow `estimated_not_allowed`
- `missing_compare_values`
- critical stale data
- policy redlist entries
- missing `metadata.ai_websearch_enhanced`
- unresolved non-fund-flow gap monitor items

Debug-only options remain debug-only:

- `--skip-gap-check`
- `--allow-fallback`

### Stage4

Add `--skip-fund-flow-check` to Stage4 with the same effective semantics as
Stage3.

Stage4 must still enforce:

- `metadata.ai_websearch_enhanced=true`
- Pring `analysis_date` matches market data date
- non-fund-flow unified quality blockers
- non-fund-flow unresolved gap monitor items
- `pring_result.fallback_used` is not true

Add an explicit debug escape hatch such as `--allow-fallback-report`. Without
that flag, Stage4 refuses to generate a formal report from a fallback Pring
result.

Direct calls to `generate_report()` are not part of the production workflow.
They remain test or compatibility usage only.

## Shared Gate Helpers

Create or extend helpers so Stage3, Stage4, audit, and report rendering use
the same filtering behavior.

Recommended helpers:

- `build_pipeline_quality_state(market_payload, stage, allow_estimated)`
- `filter_effective_quality_blockers(state, skip_fund_flow_check=False)`
- `filter_effective_gap_items(market_payload, quality_state, gap_items,
  skip_fund_flow_check=False)`
- `assert_no_fallback_pring_result(pring_payload, allow_fallback_report=False)`

`FUND_FLOW_SKIP_REASONS` should be defined once and reused by Stage3, Stage4,
and audit.

The legacy Stage2.5 `_enforce_quality_blockers()` should become a compatibility
wrapper over `build_pipeline_quality_state()` or be removed from direct tests.
It should not keep an independent implementation.

`simple_report` should consume the unified quality state for its quality
section instead of recomputing a separate issue list.

## BDI Business-Day Freshness

The current BDI allowlist uses calendar days. That blocks Friday BDI values on
Monday, even when no fresher market value is reasonably available.

Replace the operational meaning with business-day-aware freshness:

- A Friday BDI value is allowed for a Monday report.
- A Friday BDI value is blocked for a Tuesday report unless a holiday calendar
  explicitly justifies it.
- Missing dates still block.
- Untrusted domains still block.
- Unit mismatch still blocks.
- Value-range failures still block.

Initial implementation can use deterministic weekday logic without an external
calendar:

- `report_date` comes from market metadata when available.
- `entry_date` comes from `as_of_date`, `date`, or `report_period`.
- If `report_date` is Monday and `entry_date` is the immediately preceding
  Friday, freshness passes.
- Otherwise fallback to configured calendar-day max age.

Keep Stage2 BDI search freshness strict so the pipeline still tries to find
new data. The business-day exception belongs in policy allowlist evaluation,
not in search query broadening.

The policy config should make this explicit. For example:

```yaml
bdi_estimated_allow_conditions:
  trusted_domains: [...]
  max_age_days: 2
  weekend_grace: true
  value_range: [200.0, 10000.0]
  unit_keywords: ["point", "points"]
```

The existing temporary `max_age_days: 4` should not be the long-term fix.

## Audit Rule Inventory

Add a read-only inventory step that records where each quality rule comes from
and which stages consume it.

Suggested command:

```bash
bash run_clean.sh python scripts/audit_pipeline_rules.py \
  --market-data "data/runs/${DATE_NH}/market_data_complete.json" \
  --output "data/runs/${DATE_NH}/audit_rule_inventory.json"
```

Inventory output should include:

- rule id, such as `missing_source_url`
- source module, such as `pipeline_quality_state`
- affected categories
- default severity
- whether the rule is blockable
- whether `--skip-fund-flow-check` may filter it
- consuming stages, such as Stage2.5, Stage3, Stage4, report
- notes for rules delegated to Stage2.5, such as official manual override and
  fund-flow tier/window classification

This makes rule ownership reviewable before more audit checks are added.

## Audit Consistency Checker

Add a consistency checker that compares effective blockers across the pipeline
for the same input payload.

Suggested command:

```bash
bash run_clean.sh python scripts/audit_pipeline_consistency.py \
  --market-data "data/runs/${DATE_NH}/market_data_complete.json" \
  --pring-result "data/runs/${DATE_NH}/pring_result.json" \
  --gap-monitor "data/runs/${DATE_NH}/gap_monitor.json" \
  --skip-fund-flow-check \
  --output "data/runs/${DATE_NH}/pipeline_audit.json"
```

The checker computes:

- raw unified quality blockers
- Stage3 effective blockers
- Stage4 effective blockers
- effective gap monitor blockers
- report quality section issues
- fallback result status

It emits `rule_drift` when two consumers disagree without an explicit allowed
reason. Examples:

- Stage3 filters fund-flow blockers but Stage4 does not.
- report quality section flags a different blocker set than Stage4.
- legacy Stage2.5 compatibility wrapper differs from unified quality state.
- gap monitor blocks a stale item that live quality state no longer blocks.

Stage4 should read `pipeline_audit.json` when present and block on audit
`errors`. If missing, Stage4 may compute the equivalent checks inline so
formal report generation does not depend on a stale audit artifact.

## Manual Evidence Audit

Manual evidence audit covers gaps not represented by the existing quality
state. It must call existing helpers for:

- source URL parsing and official source trust
- BDI allowlist
- fund-flow source tier, window evidence, and metric basis
- Stage2.5 official manual override eligibility

It must not redefine those rules.

Suggested command:

```bash
bash run_clean.sh python scripts/audit_manual_evidence.py \
  --manual-data "data/runs/${DATE_NH}/websearch_results_manual.json" \
  --market-data "data/runs/${DATE_NH}/market_data_complete.json" \
  --stage2-log "logs/runs/${DATE_NH}/stage2_unified_log.json" \
  --output "data/runs/${DATE_NH}/manual_evidence_audit.json"
```

Hard errors:

- numeric manual entry lacks parseable URL evidence
- explicit `source_url` is not a single HTTPS URL
- `source` claims one provider while `source_url` points to another provider
- template or placeholder URL remains in a numeric entry
- evidence URL is clearly malformed
- a value claims official status but fails the existing official-source helper

Warnings:

- RRR manual value conflicts with Stage2 structured-provider value and no
  explanation is present
- `previous_value` or `change_rate` is manually supplied without a note or
  previous-source evidence
- DXY or BCOM date equals report date before the relevant market close window
- source URL is a brand landing page or index overview rather than a value
  evidence page
- source URL is reachable only as a gated page, but another evidence URL in the
  same payload supports the value

Stage4 should block on manual evidence `errors`. Warnings should be recorded in
the report or observability artifacts but should not block by default.

## 2026-05-25 Rerun Policy

After implementation, regenerate the 2026-05-25 report through formal stages:

1. Fix `websearch_results_manual.json` BCOM evidence so source and URL match.
2. Resolve RRR evidence conflict by using a supported value and an explicit
   source explanation.
3. Re-run Stage2.5 after any config or manual-data change.
4. Re-run Stage3 with only:
   - `--allow-estimated`
   - `--skip-fund-flow-check`
5. Re-run Stage4 with:
   - `--skip-fund-flow-check`
6. Confirm:
   - `pring_result.fallback_used=false`
   - report date equals Pring analysis date
   - report no longer contains `allow_fallback=TRUE`
   - only accepted estimated values are listed

## Documentation Updates

Update `AGENTS.md` and `CLAUDE.md`:

- production report generation must use Stage4 formal entry
- direct `generate_report()` usage is not a production path
- `--allow-fallback` and `--skip-gap-check` are debug-only
- accepted production downgrade is limited to fund-flow skip
- config changes require Stage2.5, Stage3, and Stage4 reruns in order
- BDI Monday freshness uses business-day semantics
- manual `source_url` must be exact evidence, not a brand placeholder
- pipeline audit commands are part of the final verification checklist

## Multi-Agent Execution Model

The implementation is suitable for parallel work if agents share contracts
first and avoid editing the same files at the same time.

### Shared Contract Phase

One coordinating agent should land or stage the shared contract before feature
agents branch off:

- `FUND_FLOW_SKIP_REASONS`
- effective blocker filter helper
- effective gap item filter helper
- fallback Pring assertion helper
- audit output schemas for `pipeline_audit.json`,
  `audit_rule_inventory.json`, and `manual_evidence_audit.json`

Until this contract exists, parallel agents should only work on tests or
design notes.

### Parallel Work Packages

Agent A: Gate parity and Stage4 hardening

- Files: `scripts/stage4_report_generator.py`,
  `scripts/stage3_pring_analyzer.py`, shared gate helper module,
  `tests/test_stage4_docs.py`, `tests/test_stage3_guard.py`
- Owns: Stage4 `--skip-fund-flow-check`, fallback result rejection, date and
  gap behavior preservation.

Agent B: BDI business-day freshness

- Files: `src/datasource/utils/policy_rules.py`,
  `config/policy_rules.yaml`, `tests/test_policy_rules.py`,
  `tests/test_stage3_guard.py`
- Owns: weekend grace semantics and policy docs for BDI.

Agent C: Audit inventory and consistency

- Files: new audit utility module, `scripts/audit_pipeline_rules.py`,
  `scripts/audit_pipeline_consistency.py`, `tests/test_pipeline_audit.py`
- Owns: rule inventory, effective blocker comparison, drift detection.

Agent D: Manual evidence audit

- Files: new manual evidence audit module,
  `scripts/audit_manual_evidence.py`, `tests/test_manual_evidence_audit.py`
- Owns: source/provider mismatch checks, placeholder URL checks, conflict
  warnings, evidence audit output.

Agent E: Report quality and documentation

- Files: `src/datasource/generators/simple_report.py`, `AGENTS.md`,
  `CLAUDE.md`, docs tests
- Owns: report quality section consuming unified state, runbook updates, final
  verification checklist.

Agent F: 2026-05-25 replay verification

- Files: run artifacts only after code is merged, no shared source edits
- Owns: rerunning Stage2.5, Stage3, Stage4 and capturing final verification
  results.

### Coordination Rules

- Each agent should work in its own git worktree or branch.
- Agent A owns the shared gate helper API. Other agents should depend on it,
  not redefine it.
- Agent C and Agent D may proceed in parallel after the audit output schemas
  are agreed.
- Agent E should update docs after Agent A and Agent B contracts are stable.
- Agent F should not start replay until A through E are merged or explicitly
  stacked in a known order.
- If two agents need the same file, the owner edits first and the second agent
  rebases or applies a narrow follow-up patch.

## Tests

Add or adjust tests:

Stage4 and Stage3:

- Stage4 with ETF-only fund-flow gaps passes when
  `--skip-fund-flow-check` is set.
- Stage4 with non-fund-flow blockers still fails even with
  `--skip-fund-flow-check`.
- Stage4 rejects `fallback_used=true` by default.
- Stage4 allows fallback reports only with explicit debug flag.
- Stage3 and Stage4 effective blockers match for the same skip options.

BDI policy:

- Monday report accepts preceding Friday BDI from trusted domains.
- Tuesday report blocks preceding Friday BDI without a holiday exception.
- untrusted BDI source blocks.
- unit mismatch blocks.
- missing BDI date blocks.

Audit:

- pipeline consistency reports no drift when Stage3 and Stage4 share the same
  effective blocker set.
- pipeline consistency reports drift when a consumer ignores the shared skip
  semantics.
- manual evidence audit errors on source/provider mismatch.
- manual evidence audit errors on placeholder or malformed URLs.
- manual evidence audit warns on RRR structured/manual conflicts.
- report quality section consumes unified quality state.
- Stage2.5 legacy quality wrapper matches `build_pipeline_quality_state()`.

Replay:

- A 2026-05-25 fixture with ETF unresolved but fund-flow skip enabled can
  generate a formal report only when `fallback_used=false`.
- The same fixture with `fallback_used=true` is blocked by Stage4.

## Acceptance Criteria

1. Formal Stage4 can generate a report with an ETF-only fund-flow gap only when
   `--skip-fund-flow-check` is explicitly set.
2. Formal Stage4 cannot generate a report from `fallback_used=true` unless a
   debug-only fallback-report flag is explicitly set.
3. Stage3 and Stage4 produce the same effective blocker set under the same
   skip options.
4. `simple_report` no longer maintains an independent blocker vocabulary.
5. BDI Friday data is accepted on Monday and blocked on Tuesday by tests.
6. Manual evidence audit catches the BCOM source/source_url mismatch pattern.
7. Pipeline audit emits no drift for a clean run and emits drift for deliberately
   inconsistent consumer behavior.
8. `AGENTS.md` and `CLAUDE.md` document the production command sequence and
   debug-only flags.
9. The regenerated 2026-05-25 report has `fallback_used=false` and no
   `allow_fallback=TRUE` marker.
