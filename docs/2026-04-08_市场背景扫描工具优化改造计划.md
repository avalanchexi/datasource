# 市场背景扫描工具优化改造计划

日期: 2026-04-08
对象: `datasource` 背景扫描/Pring/报告流水线
作者: `/autoplan` 初稿

## 1. 要解决的不是“某个脚本慢”，而是整个系统边界失真

当前仓库名义上还是“统一金融数据源框架”，但真实产品已经是一个多阶段市场背景扫描流水线：

- Stage1: API 采集
- Stage2: WebSearch + LLM 抽取增强
- Stage2.5: 手工/半手工注入补缺
- Stage3: Pring 分析与守门
- Stage4: Markdown 报告生成

问题不在于功能不够多，问题在于同一份业务真相被放在了多套机制里重复维护：

- `missing_items` 同时存在于顶层和 `metadata`
- 缺口状态同时散落在 `gap_monitor.json`、`manual_required`、`is_stale`、`is_estimated`
- Stage2、Stage2.5、Stage3、报告生成器都在重复理解“什么叫可进入下一阶段”
- README/`manager.py` 仍在描述“通用数据源管理器”，实际复杂度已经集中在脚本化流水线

这导致一个典型后果：修一个规则时，需要同时担心搜索任务生成、注入逻辑、Stage3 守门和最终报告是否会再次各自定义一遍。

## 2. 现状判断

### 2.1 现有优势

- 已经形成清晰的阶段意识，数据链路不是完全混乱
- 有较强的质量意识，存在 `policy_rules`、`quality_metrics`、`observability`、`gap_monitor`
- trend history 已经从“临时报表回填”进化为独立数据资产
- Stage2 有较完整的任务规划、搜索、抽取、冲突解决、日志链路
- 测试不是没有，关键阶段已有单元测试和部分集成测试

### 2.2 当前真正的瓶颈

- 单文件过大，导致职责耦合严重
- 状态模型重复表达，阶段间约束不一致
- 运行入口过多，主路径和遗留路径并存
- “手工补数”已经是主流程的一部分，却仍像旁路脚本
- 文档大量存在，但“当前唯一可信口径”主要靠 `AGENTS.md` 和 `CLAUDE.md`

## 3. 改造目标

### 3.1 业务目标

- 同一日期的一次扫描应当产出一个唯一、可追踪、可重跑的运行工件
- 人工补数不是例外，而是被正式建模的阶段
- 阶段失败时，系统应给出明确“卡在哪里、为什么、如何继续”，而不是靠操作者记忆口径
- 报告生成必须只消费“通过守门”的结构化结果，而不是再做额外业务判断

### 3.2 工程目标

- 把“流水线产品”从“通用数据源库”的历史外壳里拆出来
- 用单一状态模型替代重复的 `missing_items/manual_required/is_stale/gap_monitor`
- 把 Stage2 和 Stage2.5 的共享逻辑抽成可测试模块
- 用单一 CLI 入口统一 run/check/fill/report
- 把“当前标准流程”从文档约定，变成代码里的状态机约束

### 3.3 非目标

- 不是现在就全量重写所有脚本
- 不是推翻 TuShare/Tavily/DeepSeek 现有选型
- 不是引入数据库或服务化部署作为第一步
- 不是先做 UI，当前最缺的不是界面，是稳定、可解释、可维护的流水线内核

## 4. 核心前提

以下前提若不成立，方案要调整：

1. 这个工具的核心交付物不是“API SDK”，而是“每日背景扫描结果 + 报告”。
2. 操作者愿意接受“手工补数被正式建模”，而不是继续作为脚本旁路。
3. 单日运行的一致性比“任意阶段单独脚本可自由执行”更重要。
4. 当前项目的主要风险不是吞吐量不够，而是规则漂移、状态不一致、维护成本过高。
5. 短期内仍接受本地文件作为运行态存储，不把服务化或数据库化当作首要目标。

## 5. 北极星架构

### 5.1 一个日期，一份运行包

为每次运行建立统一运行目录，例如：

`runs/YYYYMMDD/`

其中至少包含：

- `raw/stage1_market_data.json`
- `stage2/tasks.jsonl`
- `stage2/results.json`
- `stage2.5/manual_patch.json`
- `artifacts/market_data_complete.json`
- `artifacts/pring_result.json`
- `artifacts/report.md`
- `artifacts/run_manifest.json`

`run_manifest.json` 作为唯一真相，记录：

- 当前阶段
- 阶段状态
- 阻断原因
- 未解决缺口
- 允许放行项
- 数据来源摘要
- 产物路径

这样 Stage3、Stage4 不再自己猜文件名和状态，而是只读 manifest。

### 5.2 统一状态模型

定义 `PipelineStateContract`，把所有守门状态收口为一个结构：

- `gaps.pending`
- `gaps.manual_required`
- `quality.blockers`
- `quality.warnings`
- `freshness.stale_items`
- `estimation.allowlisted`
- `estimation.blocked`
- `phase_status.stage1/stage2/stage2_5/stage3/stage4`

顶层 `missing_items`、`metadata.missing_items`、外部 `gap_monitor.json` 不再并存。它们都退化为 manifest 内部字段或其导出视图。

### 5.3 模块拆分原则

Stage2 与 Stage2.5 必须拆成 6 个稳定子模块：

1. `task_planning`
2. `search_execution`
3. `extraction_normalization`
4. `state_writeback`
5. `quality_gate`
6. `artifacts_and_logs`

原则：

- 搜索模块不直接修改 `market_payload`
- 写回模块不直接做策略判断
- 守门模块不重复做抽取或补数
- 报告生成器不再承担质量清洗与兜底判断

### 5.4 单一 CLI

目标入口：

```bash
python -m datasource.pipeline run --date 2026-04-08
python -m datasource.pipeline inspect --date 2026-04-08
python -m datasource.pipeline patch --date 2026-04-08 --from reports/websearch_results_20260408_manual.json
python -m datasource.pipeline report --date 2026-04-08
```

脚本仍可保留兼容层，但只做薄封装。

## 6. 分阶段改造方案

### Phase A, 先收口运行模型，不动业务规则

目标:

- 引入 `runs/YYYYMMDD/` 目录和 `run_manifest.json`
- 新建统一 orchestrator
- 让现有 Stage1/2/2.5/3/4 继续工作，但所有产物都由 orchestrator 编排

具体动作:

- 新增 `src/datasource/pipeline/orchestrator.py`
- 新增 `src/datasource/models/pipeline_manifest.py`
- 把当前脚本入参/出参全部改为可从 manifest 读取
- `run_background_scan_pipeline.py` 升级为真正的主入口，而不是 legacy helper

验收:

- 操作者只需要一个日期，就能重跑、补数、继续
- 阶段失败后能直接打印 manifest 摘要

### Phase B, 消灭重复状态表达

目标:

- 把 `missing_items/manual_required/is_stale/is_estimated/gap_monitor` 的判断源统一到一个地方

具体动作:

- 抽出 `src/datasource/pipeline/state_machine.py`
- 抽出 `src/datasource/pipeline/gate_rules.py`
- Stage2 写回只返回标准化 `TaskOutcome`
- Stage2.5 只消费标准化缺口，不再自己二次推断“哪里算补齐”
- Stage3 只读 manifest 的 gate 结果，不再自己重复扁平化多套状态

验收:

- Stage3 守门不再自己重新解释顶层/metadata/gap_monitor
- “白名单放行”和“compare_gaps 阻断”这类冲突必须在统一 gate 中定义优先级

### Phase C, 拆掉 Stage2 / 注入器大文件

目标:

- `scripts/stage2_unified_enhancer.py` 和 `inject_websearch_data_test.py` 分解为可读、可测的模块

具体动作:

- Stage2 拆到 `src/datasource/stage2/`
- 注入器拆到 `src/datasource/stage2_patch/`
- 复用的逻辑统一下沉：
  - source_url 校验
  - manual_required 推导
  - trend history 回填
  - quality blockers
  - gap summary

验收:

- 单模块文件控制在 400-600 行以内
- Stage2/Stage2.5 共享逻辑不再复制

### Phase D, 契约升级与兼容层治理

目标:

- 数据契约版本化
- 区分“核心产物字段”和“运行中间态字段”

具体动作:

- `MarketDataContract` 增加 `schema_version`
- 把运行态字段逐步迁出业务实体，避免 `MacroIndicatorData` 同时承载数据、运行状态、补数痕迹
- 给 legacy key、legacy task、legacy source 建单独兼容层，避免散落在业务代码

验收:

- Stage2/Stage3/Report 之间靠版本化契约交互
- legacy 兼容逻辑能集中搜索到，不再全仓库散布

### Phase E, 把报告生成器从“半个守门器”变回纯视图层

目标:

- 报告生成器只负责展示，不再负责业务纠错和状态修复

具体动作:

- `simple_report.py` 只消费已通过 gate 的完整产物
- 质量日志、allow_estimated、trend_history 修补从报告层移出
- 为报告层提供一个已归一化的数据视图 DTO

验收:

- 报告生成失败只可能是模板/字段缺失问题，不应该再暴露业务守门冲突

### Phase F, 开发者体验收敛

目标:

- 项目对维护者友好，不再依赖阅读长篇操作文档才能执行

具体动作:

- 保留 `run_clean.sh`，但把常用命令统一到 `datasource pipeline`
- README 改写为“背景扫描产品”而不是“通用数据源框架”
- 为常见问题输出机器可读诊断
- 提供 `datasource pipeline doctor --date ...`

验收:

- 新维护者 10 分钟内能理解主路径
- 不需要同时看 `README`、`CLAUDE.md`、`AGENTS.md` 才能知道应该跑哪个命令

## 7. 测试策略

### 7.1 必补测试

- 统一 gate 优先级测试
- manifest 状态机测试
- Stage2 写回与 Stage2.5 注入共享规则测试
- golden dataset 回归测试
- 单日全链路 smoke test
- 失败恢复测试，尤其是“Stage2 失败 → 补数 → 继续 Stage3”

### 7.2 应删或降级的测试形态

- 过度依赖内部私有函数名的测试
- 对 legacy 兼容细节的重复断言
- 同一个规则在 Stage2、注入器、Stage3 重复测三遍但口径还不一致

## 8. 风险与应对

### 风险 1, 改造期双轨并存更复杂

应对:

- 明确只保留一个新主入口
- 老脚本全部变为 thin wrapper

### 风险 2, 历史数据文件格式不一致

应对:

- 引入 schema_version
- 提供一次性 migration 脚本

### 风险 3, 规则收口时暴露更多真实缺口

应对:

- 接受短期内“更严格但更吵”
- 先把规则一致性做对，再谈减少阻断噪音

## 9. 成功指标

- 主入口数量从多个收敛到 1 个
- 单次运行状态查看从“读多个 JSON + 文档”降到“读一个 manifest”
- Stage2/Stage2.5/Stage3 的重复规则代码下降 50%+
- 关键热点文件不超过 600 行
- 单日全链路 smoke test 稳定通过
- 新维护者首次成功跑通时间 < 15 分钟

## 10. 立即执行顺序

### P0, 先做

- 建 `run_manifest.json` 与 orchestrator
- 统一缺口/阻断状态模型
- 修 Stage3 gate 规则优先级冲突
- 收口 Stage2 与 Stage2.5 的共享规则

### P1, 接着做

- 拆 Stage2 大文件
- 拆注入器大文件
- 建黄金样本回归测试

### P2, 再做

- README/CLI/doctor 收敛
- 契约版本化
- 报告层纯视图化

## 11. 不做的事

- 现在不做 Web UI
- 现在不把它服务化成在线系统
- 现在不重写 Pring 核心算法
- 现在不追求“所有旧脚本立刻消失”，但必须让它们退出主路径

## 12. 结论

这项目最该改的不是某个算法，而是“运行真相放在哪里”。

如果继续沿着现在的方式加规则，半年后你会得到一个很会工作但没人敢动的系统。它每天都能出报告，但每次修一处都像拆炸弹。

正确路线不是推翻重来，而是先把运行态、状态机、主入口收口，再去拆大文件。先把系统边界做出来，后面的重构才是线性的。
