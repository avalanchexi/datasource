## 一、V3.1 当前限制分析

### 1.1 核心问题

**问题描述**: V3.1三阶段架构生成的报告包含大量占位符数据（"N/A"、"待MCP获取"），数据完整度仅38%。

**影响范围**:
- ❌ **商品数据** (6/6 占位符): COMEX黄金、WTI原油、Brent原油、COMEX铜、BCOM指数、GSG ETF
- ❌ **债券数据** (2/3 占位符): CN10Y、CN10Y_CDB
- ❌ **资金流向** (4/4 占位符): 北向资金、南向资金、ETF资金流、融资融券
- ❌ **财经要闻** (0/12): 完全缺失

**根本原因**: V3.1严格遵循单一职责原则，将MCP数据获取逻辑**完全排除**在三阶段架构之外，以保持架构纯净性。

### 1.2 设计决策回顾

V3.1采用**占位符模式** (Placeholder Pattern)：

```python
# Stage 1: 数据收集器
- 职责: 仅收集Traditional API数据（TuShare/AKShare/Yahoo Finance）
- 策略: 为不可用数据创建占位符，标记"待MCP获取"
- 输出: MarketDataContract (38% 完整度)

# Stage 2: Pring分析器
- 职责: 基于可用数据进行Pring三层分析
- 策略: 处理不完整数据，标记WebSearch需求
- 输出: PringResultContract (分析结果 + 待补充字段列表)

# Stage 3: 报告生成器
- 职责: 纯模板渲染，无业务逻辑
- 策略: 直接输出占位符，不调用MCP工具
- 输出: Markdown报告 (包含占位符)
```

**这是有意为之的架构设计**，而非缺陷。

### 2.2 Stage 4 职责定义

**输入**:
1. `market_data.json` - 包含占位符的市场数据
2. `pring_result.json` - Pring分析结果（含待补充字段）
3. `report_draft.md` - 包含占位符的报告草稿

**核心职责（当前实现）**:
1. **扫描占位符**: 聚焦资金流向（北向/南向/ETF/融资融券）与财经要闻
2. **MCP工具调用**: 优先MCP WebSearch，失败时自动降级AKShare并记录来源
3. **数据验证**: 检测异常零值并生成“异常零值-需核查”备注
4. **占位符替换**: 更新MarketDataContract（fund_flow、financial_news）；可选触发Stage 3再渲染
5. **完整性报告**: 输出`logs/YYYYMMDD_stage4_log.json`记录填充细节

**输出**:
1. `market_data_enhanced.json` - 更新后的市场数据 (资金流向+要闻覆盖，95%+完整度)
2. `report_enhanced.md` - （可选）重新生成的最终报告 (`--regenerate-report`)
3. `enhancement_log.json` - Stage 4日志（来源追溯/异常记录）

## 四、临时解决方案（V3.1 → V3.2 过渡期）

### 4.1 AI手工填充工作流

**适用场景**: V3.2未完成前，需要生成完整报告

**步骤**:
1. 使用V3.1生成草稿报告
2. AI读取报告，识别占位符
3. AI使用MCP工具手工填充数据
4. AI更新报告并验证

**优势**:
- ✅ 立即可用
- ✅ 保持V3.1架构纯净性
- ✅ 验证Stage 4设计可行性

**劣势**:
- ❌ 需要人工/AI介入
- ❌ 无法自动化
- ❌ 耗时较长（10-15分钟）

### 4.2 使用V2.1作为备选

**何时使用V2.1**:
- 生产环境急需完整报告
- V3.2尚未部署
- 无法进行AI手工填充

**V2.1执行**:
```bash
PYTHONIOENCODING=utf-8 python scripts/utility/background_scan_120d_generator.py \
    --date 2025-11-11 \
    --output reports/20251111背景扫描120.md
```

**注意**: V2.1为单体架构，缺乏V3.1的可测试性和可维护性优势。

- ✅ 架构纯净性（单一职责原则）
